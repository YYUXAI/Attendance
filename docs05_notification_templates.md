# 通知模板约定（template_id）

⚠️ 本文件不是唯一真相来源

优先级：
1. docs00_system_truth.md（最高）
2. 数据库 DDL
3. 本文件

---

# 一、作用域（强约束）

本文件仅定义：

👉 进入 notification_queue 的“异步通知任务模板”

不包含：

❌ handler 内的 message.reply / callback.answer  
❌ 表单输入错误提示  
❌ 按钮交互提示  
❌ 群内即时回执（如打卡成功/失败）  
❌ /start、菜单引导等交互文案  

原则：

👉 notification_queue 是“异步任务队列”，不是“统一消息出口”

---

# 二、template_id 定义

template_id 表示：

👉 “通知任务类型（业务枚举）”

它是：

- ✅ 稳定业务语义标识
- ✅ 幂等约束组成部分
- ✅ 统计维度

它不是：

- ❌ 文案模板 ID
- ❌ 动态配置项
- ❌ 渲染指令

---

# 三、与数据库关系（强约束）

notification_queue 唯一约束：

(log_id, notify_tg_id, template_id) UNIQUE

含义：

👉 同一事件 + 同一接收人 + 同一通知类型，只允许一条通知

因此：

⚠️ template_id 必须稳定且不可复用语义

---

# 四、template_id 与 reply_content（强约束）

template_id：标识“通知类型”  
reply_content：实际发送内容  

规则：

1. worker 只发送 reply_content
2. 禁止根据 template_id 拼装文案
3. reply_content 必须在入队时完整生成
4. template_id 不参与内容生成

结论：

👉 notification_queue 是“发送执行层”，不是“模板渲染层”

---

# 五、通知判定规则（强约束）

一个消息必须满足以下条件，才允许进入 notification_queue：

- 不依赖当前交互即时返回
- 允许异步发送
- 需要失败重试
- 需要记录送达结果
- 需要幂等控制
- 属于业务事件后的触达行为

否则：

👉 必须在 handler / service 中直接 reply，不得入队

---

# 六、编码规则（强约束）

1000-1999：审批模块  
2000-2999：质检模块  
3000-3999：审计模块  
9000-9999：系统通知  

规则：

1. 必须使用整数
2. 不允许跨模块复用
3. 不允许字符串 template_id
4. 不允许动态生成 template_id

# 六点补充：注册表边界

本文件只定义：

👉 template_id 的稳定业务语义

本文件不定义：

- 某条通知在当前版本中是否一定生成
- 某条通知当前是否已经实现
- 某条通知当前是否允许补建

这些应以 docs00_system_truth.md 和具体业务实现为准。

## 质检模块（2000）

2001 QC_DRAW_NOTICE  
👉 抽检通知（面向被抽中员工的触达语义；不得复用为「考勤群轮次开始公告」或「班次级汇总」）

2002 QC_TIMEOUT_NOTICE  
👉 超时提醒  

2003 QC_RESULT_NOTICE  
👉 质检结果通知（面向结果触达的通用语义；不得复用为「每轮完结带交互按钮的群公告」或「班次结束后整班次质检汇总公告」）

2004 QC_ROUND_START_GROUP_NOTICE  
👉 每轮质检开始 — 考勤群公告  
说明：
- 进入 `notification_queue` 时使用本 `template_id`
- 异步发送、可重试、可幂等（与 `docs05` 第五节入队条件一致）
- 稳定语义：某一质检轮次在考勤群内的「本轮开始」告知，与 2001 的「抽检通知」区分

2005 QC_SHIFT_QC_SUMMARY_GROUP_NOTICE  
👉 班次结束后质检汇总 — 考勤群公告（例如班次结束后固定延迟触发的整班次质检汇总）  
说明：
- 进入 `notification_queue` 时使用本 `template_id`
- 与 2003「质检结果通知」区分：本 id 专指**班次维度**汇总触达，不得套用 2003 表达新语义

### 质检群公告入队边界（与模板职责）

以下三类须区分是否进入 `notification_queue`（`reply_content` 入队时须已完整生成；本节不讨论「点击查看截图」后的群内发图，该行为不属于 `notification_queue`）：

1. **每轮质检开始公告（考勤群）**  
   👉 **应进入** `notification_queue`，使用 **2004**。

2. **每轮质检完结公告（考勤群，带【查看截图】等 inline 按钮）**  
   👉 **不进入** `notification_queue`。  
   原因：`notification_queue` 在本文档约束下以 `template_id + reply_content` 驱动 worker 发送为主；**带固定 inline 键盘/回调负载的群消息**属于强交互编排，通常需与 callback 路由同路径维护，且与本文档「未纳入」的交互类消息边界一致。该条公告应由业务服务/机器人在**非队列路径**发送（仍可用 `file_id` 直发，不生成下载链接）。
   边界约束：
   - **不得因未送达任务或异常中间状态导致该轮完结公告长期不触发**；进入本轮 `qc_task_queue` 的任务必须在固定 15 分钟硬期限后收口，未送达也视为未完成，从而保证本轮具备可完结条件。

3. **班次结束后固定延迟的整班次质检汇总公告（考勤群）**  
   👉 **应进入** `notification_queue`，使用 **2005**。

## 审计模块（3000）

3001 AUDIT_ABNORMAL_NOTICE  
👉 异常审计通知  
说明：
- 用于审计阶段发现异常结果后的异步触达
- 属于“异常提醒”语义
- 不等同于开班汇总、下班提醒、负责人汇总

3002 AUDIT_SUMMARY_NOTICE  
👉 审计汇总通知  
说明：
- 用于审计结果的汇总类通知
- 属于通用“汇总通知”语义
- 不得复用为新的细分稳定通知类型

3003 AUDIT_SHIFT_START_GROUP_NOTICE  
👉 开班考勤群公告  
说明：
- 用于班次上班时间到达时，发往考勤群的开班汇总公告
- 典型内容包括：
  - 日期
  - 部门
  - 班次
  - 时区
  - 部门负责人
  - 应到岗人数
  - 已到岗人数
  - 报备休假人数及名单
  - 迟到人数及名单
  - 未打卡人数及名单
  - 质检即将启动提醒
- 属于“班次开班公告”稳定语义
- 不得复用为组长私信、部门负责人汇总或下班提醒

3004 AUDIT_SHIFT_START_LEADER_DM_NOTICE  
👉 开班组长私信通知  
说明：
- 用于班次上班时间到达时，发给当前组织组长的私信提醒
- 典型内容包括：
  - 班次已开始
  - 应到岗人数
  - 已到岗人数
  - 报备休假人数及名单（如有）
  - 上班时间已到但仍未打卡人员名单（如有）
- 属于“组长关注待确认员工状态”的稳定语义
- 不得复用为考勤群公告或部门负责人汇总

3005 AUDIT_SHIFT_START_DEPARTMENT_HEAD_NOTICE  
👉 开班部门负责人汇总私信  
说明：
- 用于班次上班时间到达时，发给部门负责人的跨组汇总私信
- 典型内容包括：
  - 部门
  - 班次
  - 时区
  - 各组应到岗人数
  - 各组实际到岗人数
  - 各组报备休假人数
  - 各组未打卡人数
- 属于“部门负责人总览多个组状态”的稳定语义
- 该 template_id 可以先定义并保留
- 具体业务层是否生成该通知，取决于系统内是否存在可靠的“部门-组聚合关系”数据来源
- 禁止因为当前数据结构不足而修改该 template_id 的业务语义

3006 AUDIT_SHIFT_END_GROUP_REMINDER_NOTICE  
👉 下班考勤群提醒公告  
说明：
- 用于班次下班时间到达时，发往考勤群的统一提醒
- 典型内容包括：
  - 日期
  - 部门
  - 班次
  - 时区
  - 工作时间已结束
  - 请勿忘记打卡
- 属于“下班提醒”稳定语义
- 不得复用为开班公告或异常通知

# 八、当前未纳入（明确排除）

以下场景属于“交互消息”，不得使用 template_id：

- 注册流程所有提示（成功/失败/重复）
- /start 菜单
- 按钮点击提示
- 输入校验错误提示
- 打卡成功/失败回执
- handler 内任何 message.reply

原因：

👉 这些消息不具备“任务属性”，不需要进入 notification_queue

---

# 九、补建与重试规则（强约束）

必须区分两类情况：

## 1. 通知任务缺失补建

定义：

👉 某业务事件本应写入 notification_queue，但由于系统原因未成功写入，导致“应有通知任务缺失”

规则：

- 允许业务层在后续巡检中补建缺失通知任务
- 补建的前提是：该业务语义对应的通知任务记录不存在
- 补建属于“任务生成层”的职责

---

## 2. 通知发送失败重试

定义：

👉 通知任务已经成功写入 notification_queue，但发送时失败

规则：

- 不允许再次补建新的通知任务
- 必须交由 notification worker 按既有状态机进行重试
- 重试属于“任务执行层”的职责

---

## 3. 结论

必须遵守：

- 缺失补建 ≠ 发送重试
- 已入队任务发送失败，不得通过“再插一条新通知”解决
- notification_queue 负责发送执行与送达结果
- 业务模块只负责“该不该生成这条通知任务”

---

# 十、扩展规则（强约束）

新增 template_id 必须满足：

1. 是新的“通知任务类型”，不是文案变化
2. 不允许复用旧编号表达新语义
3. 不允许删除已使用编号
4. 必须同步更新本文件
5. 如代码中采用常量管理，常量名必须与本文件语义一致

注意：

👉 注册一个 template_id，不等于当前版本一定会生成该通知
👉 是否生成，取决于业务规则与实现阶段

# 十一、禁止行为

禁止：

- 在代码中写 template_id = NULL
- 在代码中硬编码未定义 template_id
- 使用 template_id 生成文案
- 将交互消息错误入队
- 修改 template_id 语义但不更新文档
- 将“通知缺失补建”和“发送失败重试”混为一谈
- 用 3001 / 3002 复用表达 3003 / 3004 / 3005 / 3006 的新语义

---

# 十二、设计原则总结

必须遵守：

1. 通知模块只处理“异步通知任务”
2. template_id 是“业务语义”，不是“文案模板”
3. notification_queue 是执行层，不是模板系统
4. 交互消息与通知任务必须分离
5. 所有通知必须可追溯、可幂等、可失败重试
6. 缺失补建与发送重试必须分层处理
